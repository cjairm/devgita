# Cycle: `dg list` — show installed packages by category

**Date:** 2026-07-06
**Estimated Duration:** ~2–3 hours
**Status:** Done

---

## 1. Domain Context

Devgita tracks everything it installs in `~/.config/devgita/global_config.yaml` (product
principle #6, [CLAUDE.md](../../../CLAUDE.md) §3). Today there is no way to _see_ that state
short of `cat`-ing the YAML file by hand. [ROADMAP.md](../../../ROADMAP.md) lists `dg list` /
`dg installed` as a planned command: "Display installed packages with versions and timestamps;
show category grouping."

**Scope note (read before implementing):** the current `GlobalConfig` schema stores installed
items as bare name strings per category bucket — there is no per-item version or install-timestamp
field anywhere in the config (confirmed by reading `internal/config/fromFile.go`; the only
timestamp in the whole file is `FailedInstallation.FailedAt`, which is for install _failures_, not
successes). Adding version/timestamp tracking means changing the config schema, which
[CLAUDE.md](../../../CLAUDE.md) §10 explicitly calls out as needing a migration strategy and
explicit discussion — not something to fold silently into a "show me what's installed" cycle.

**Decision (confirmed with user 2026-07-06):** this cycle ships the MVP — name + category only,
against the existing schema, no config format change. Versions/timestamps are an explicit
follow-up cycle once the schema change is scoped on its own.

Also worth noting: `cmd/root.go`'s help `Long` text already advertises `list` (and
`check-updates`, `backup`, `restore`, `validate`, `change`, `update`, `reinstall`,
`re-configure`) as "Available Commands" — none of these exist as registered commands today. This
cycle makes `list` true; the rest of that stale help text is out of scope here but should be
flagged/fixed in a follow-up (either implement or trim the help text to match reality).

## 3. Objective

Implement `dg list` (alias `dg installed`), a read-only command that prints everything Devgita
has installed, grouped by category, plus a separate section for items that were already present
on the machine before Devgita touched them — sourced entirely from the existing
`global_config.yaml` schema, no format changes.

## 4. Scope Boundary

### In Scope

- [x] `dg list` command (`Aliases: []string{"installed"}`) registered via its own `init()`
      (mirroring `cmd/task.go`)
- [x] Reads `GlobalConfig.Installed` and prints one table per non-empty category
      (Packages, Desktop Apps, Fonts, Themes, Terminal Tools, Dev Languages, Databases)
      using `text/tabwriter`, matching the `cmd/worktree.go` list style
- [x] Separate "Already on this machine (not installed by Devgita)" section from
      `GlobalConfig.AlreadyInstalled`, only printed if non-empty
- [x] `--category <name>` flag to filter to a single bucket (accepts the yaml key names:
      `packages`, `desktop_apps`, `fonts`, `themes`, `terminal_tools`, `dev_languages`,
      `databases`); invalid value → clear error listing valid names
- [x] Handles the empty-config case (nothing installed yet) with a clear message, not a blank
      screen or panic
- [x] Tests for the formatting function (empty config, single category, multiple categories,
      `--category` filter, invalid `--category`) and for the config-load-failure path
- [x] Document `dg list` in `docs/spec.md` and README

### Explicitly Out of Scope

- Per-item version tracking — requires a `GlobalConfig` schema change + migration; deferred to a
  future cycle (see Domain Context above)
- Per-item install timestamp — same reason
- `dg check-updates` (finding available updates) — separate ROADMAP item, depends on version
  tracking existing first
- Fixing the stale `cmd/root.go` help text that advertises unimplemented commands — flagged here,
  not fixed in this cycle
- Any change to `internal/config/fromFile.go`'s schema or `Load`/`Save` behavior

**Scope is locked.** If something out of scope surfaces, document it here for a future cycle.

## 5. Implementation Plan

### File Changes

| Action | File Path          | Description                                                                        |
| ------ | ------------------ | ---------------------------------------------------------------------------------- |
| Create | `cmd/list.go`      | `listCmd` + `--category` flag + `formatInstalled()` helper + registration `init()` |

| Modify | `docs/spec.md`     | Document `dg list` under Features                                                  |
| Modify | `README.md`        | Add `dg list` to command list, matching the `dg task`/`dg worktree` entry style    |

### Step-by-Step

#### Step 1: Write `formatInstalled()` as a pure function

- In `cmd/list.go`, add `formatInstalled(gc *config.GlobalConfig, category string) (string, error)`
  that takes the loaded config and an optional category filter (`""` = all) and returns the
  rendered tables as a string (easier to unit test than writing straight to `os.Stdout`).
- Category name → `InstalledConfig` field mapping is a simple switch/map keyed by the yaml tag
  names listed above.
- Verify: `go build ./cmd/`

#### Step 2: Test `formatInstalled()`

- Cover: empty `GlobalConfig{}` (message, not blank/panic), one category populated, several
  categories populated (order should be stable — iterate a fixed slice of category names, not a
  map), `--category=fonts` filter, unknown category name → error.
- Verify: `go test ./cmd/ -run TestFormatInstalled -v`

#### Step 3: Wire up `listCmd`

- `Use: "list"`, `Aliases: []string{"installed"}`, `RunE` loads `config.GlobalConfig{}` via
  `.Load()` (wrap load errors per the `cmd/uninstall.go` pattern), calls `formatInstalled`, prints
  result.
- `--category` flag via `StringVar`.
- Register in `cmd/list.go`'s own `init()`: `rootCmd.AddCommand(listCmd)`.
- Verify: `go build ./cmd/ && ./devgita list --help`

#### Step 4: Command-level tests

- `cmd/list_test.go`: verify flag parsing and that an invalid `--category` value surfaces the
  expected error message.
- Verify: `go test ./cmd/ -v`

#### Step 5: Manual verification against a real config

- Run against a machine with an actual `~/.config/devgita/global_config.yaml` (or a temp one via
  `HOME` override) to confirm real output reads cleanly, both with and without `--category`.

#### Step 6: Documentation

- Add `dg list` to `docs/spec.md` Features section and to `README.md`'s command list (same section
  where `dg task` and `dg worktree` are documented).
- Verify: re-read for clarity.

## 7. Risks & Trade-offs

| Risk                                                                                | Likelihood | Mitigation                                                                                                            |
| ----------------------------------------------------------------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------- |
| Users expect versions/timestamps per ROADMAP wording and are surprised by MVP scope | Med        | Call this out explicitly in `docs/spec.md` and README as "planned: version/timestamp tracking in a future release"    |
| `--category` flag naming (yaml keys like `terminal_tools`) is not obvious to type   | Low        | `--help` and error message list valid values; alias-free for now, can add short names later if it's a real pain point |
| Map iteration non-determinism making output flaky/hard to test                      | Low        | Iterate a fixed, ordered slice of category names, never range over a Go map for output order                          |

### Trade-offs Made

- **MVP (names + category) vs. full scope (+ version/timestamp):** chose MVP per explicit user
  decision on 2026-07-06 — avoids an un-scoped config migration inside what should be a small,
  low-risk read-only command.
- **Pure formatting function vs. writing directly to stdout in `RunE`:** chose the pure-function
  split so tests don't need any I/O mocking at all — this command touches no external commands or
  files beyond the one `Load()` call.
- **`--category` flag now vs. deferring:** included because ROADMAP explicitly calls out "show
  category grouping" and it's a small addition once `formatInstalled` exists anyway.
